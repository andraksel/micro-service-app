# Пошаговый план обучения ручному QA на microservices-qa-lab

Цель: после прохождения плана уметь уверенно тестировать микросервисное backend-приложение вручную, понимать архитектуру, находить дефекты, объяснять причины ошибок, пользоваться логами, RabbitMQ, Redis, Prometheus/Grafana и уверенно отвечать на техническом собеседовании.

Автоматизация в этом плане намеренно не используется. `pytest` можно запускать только как контроль состояния стенда, но не как основной способ обучения.

## 0. Что должно получиться на выходе

После прохождения плана ты должен уметь:

- читать схему микросервисов и объяснять поток запроса от UI/API Gateway до БД и очереди;
- тестировать REST API руками через UI, curl или Postman;
- проектировать позитивные, негативные, граничные и интеграционные проверки;
- проверять асинхронные события через RabbitMQ и notification-service;
- проверять Redis cache через заголовки `X-Cache`;
- работать с Docker Compose: запуск, остановка, перезапуск сервиса, просмотр логов, reset окружения;
- анализировать JSON-логи по `X-Request-ID`;
- пользоваться Prometheus и Grafana для базовой диагностики latency, error rate, request rate;
- отличать дефект frontend, gateway, backend-сервиса, БД, кеша, очереди и внешней зависимости;
- оформлять тест-кейсы, чек-листы, баг-репорты, тестовый отчет и риск-анализ;
- отвечать на вопросы по API, микросервисам, Docker, observability, очередям, кешу и тест-дизайну.

## 2. Правила обучения

На каждом шаге веди рабочий журнал. Формат простой:


Не ограничивайся "работает / не работает". Сильный тестировщик объясняет, почему работает, где может сломаться и как это доказать.

Артефакт этапа: короткий чек-лист "Проверка готовности стенда".

## 5. Этап 3: базовое ручное API-тестирование

Цель: уверенно тестировать HTTP API.

Проверь для каждого endpoint:

- HTTP method;
- URL;
- request body;
- response body;
- status code;
- headers;
- ошибки валидации;
- повторный запрос;
- несуществующий ID;
- некорректный тип данных;
- пустые поля;
- граничные значения.

Минимальный набор ручных проверок:

| Проверка | Ожидаемый результат |
| --- | --- |
| Создать валидного пользователя | `201 Created`, статус `active`. |
| Создать пользователя с плохим email | `422 Unprocessable Entity`. |
| Создать пользователя с тем же email | `409 Conflict`. |
| Получить существующего пользователя | `200 OK`. |
| Получить несуществующего пользователя | `404 Not Found`. |
| Создать валидный продукт | `201 Created`. |
| Создать продукт с отрицательной ценой | Ошибка валидации. |
| Создать заказ с валидными user/product | `201 Created`. |
| Создать заказ с несуществующим user | Ошибка бизнес-валидации. |
| Создать заказ с quantity больше stock | Ошибка бизнес-валидации. |

Что нужно уметь объяснить:

- Разницу между `400`, `404`, `409`, `422`, `500`, `502`, `503`, `504`.
- Почему `422` часто означает ошибку схемы запроса.
- Почему `409` подходит для конфликта уникальности email.
- Почему не все ошибки являются багами.

Артефакт этапа: таблица тест-кейсов по API.

## 12. Этап 10: Prometheus и Grafana для QA

Цель: использовать метрики как инструмент тестировщика.

Открой:

```text

```

PromQL для практики:

```promql
sum by (service) (rate(http_requests_total[1m]))
sum by (service, status_code) (rate(http_requests_total[1m]))
```

Практика:

- Включи payment timeout и посмотри latency.

Что нужно уметь объяснить:

- Что такое metric, label, scrape, target.
- Чем counter отличается от histogram.
- Что такое latency p95.
- Почему QA должен смотреть не только response body, но и метрики.
- Как метрики помогают увидеть деградацию без явного падения теста.

## 16. Вопросы и ответы для собеседования

### Общие QA вопросы

**Вопрос: Что ты тестировал в этом проекте?**  
Ответ: Я тестировал микросервисный backend: user, product, order, payment и notification services. Проверял REST API, интеграционный order flow, Redis cache, RabbitMQ events, gateway routing, readiness, structured logs и метрики в Prometheus/Grafana.

**Вопрос: Какие виды тестирования ты применял?**  
Ответ: Smoke, functional, negative, boundary, integration, contract thinking, cache testing, async/event testing, resilience testing и observability-based checks.

**Вопрос: Чем checklist отличается от test case?**  
Ответ: Checklist фиксирует, что нужно проверить, обычно кратко. Test case содержит точные preconditions, steps, expected result и test data. Checklist быстрее для exploratory/regression, test case лучше для повторяемости.

**Вопрос: Чем severity отличается от priority?**  
Ответ: Severity показывает влияние дефекта на систему или пользователя. Priority показывает срочность исправления для бизнеса или релиза. Например, typo на главной странице может быть low severity, но high priority перед публичным релизом.

**Вопрос: Что такое smoke testing?**  
Ответ: Быстрая проверка, что основная функциональность системы жива и можно продолжать более глубокое тестирование. В этом проекте smoke: gateway ready, сервисы ready, create user, create product, create order.

**Вопрос: Что такое regression testing?**  
Ответ: Проверка, что уже работавшая функциональность не сломалась после изменений. Например, после добавления Grafana я проверяю, что user/product/order/payment/notification flow все еще работает.

### API и HTTP

**Вопрос: Что ты проверяешь в API response?**  
Ответ: Status code, response body, schema, required fields, business values, headers, error format, response time и побочные эффекты в других сервисах.

**Вопрос: Разница между `400` и `422`?**  
Ответ: `400` обычно означает некорректный запрос на уровне бизнес-логики или общего формата. `422` в FastAPI часто означает, что JSON синтаксически принят, но не прошел валидацию схемы: неправильный тип, отсутствующее обязательное поле, плохой email.

**Вопрос: Когда использовать `404`, а когда `409`?**  
Ответ: `404` - ресурс не найден. `409` - конфликт текущего состояния, например попытка создать пользователя с email, который уже существует.

**Вопрос: Что означает `502`?**  
Ответ: Gateway или сервис-посредник получил некорректный ответ от upstream dependency. В микросервисах это часто означает проблему при вызове другого сервиса.

**Вопрос: Что означает `503`?**  
Ответ: Service unavailable. Сервис или его зависимость временно не готовы. Readiness может возвращать `503`, если база, Redis или RabbitMQ недоступны.

**Вопрос: Что означает `504`?**  
Ответ: Timeout при ожидании ответа от зависимости. В этом проекте timeout payment-service должен приводить к контролируемому поведению order-service.

### Микросервисы

**Вопрос: Почему у каждого сервиса своя база данных?**  
Ответ: Это снижает связность и сохраняет ownership данных. Сервис владеет своей моделью и API. Другие сервисы не должны напрямую читать его БД, иначе появится сильная зависимость от внутренней схемы.

**Вопрос: Как тестировать микросервисный flow?**  
Ответ: Нужно проверять не только один endpoint, но и всю цепочку: входной запрос, вызовы зависимостей, состояние данных, события в очереди, итоговую notification, логи и метрики.

**Вопрос: Как локализовать дефект в микросервисах?**  
Ответ: Сначала фиксирую request/response и `X-Request-ID`, потом смотрю gateway logs, логи сервиса-оркестратора, логи зависимостей, readiness, RabbitMQ/Redis и метрики. Цель - определить, ошибка в клиентском запросе, gateway, конкретном сервисе или зависимости.

**Вопрос: Что такое service readiness?**  
Ответ: Это готовность сервиса принимать рабочий трафик. Она должна учитывать критичные зависимости: БД, Redis, RabbitMQ. Сервис может быть живым, но не ready.

**Вопрос: Что такое service health?**  
Ответ: Это базовая проверка, что процесс жив. Health обычно проще readiness и не всегда проверяет все зависимости.

### Docker

**Вопрос: Зачем Docker Compose в этом проекте?**  
Ответ: Он поднимает все сервисы и зависимости локально одной командой: микросервисы, PostgreSQL, Redis, RabbitMQ, nginx, Prometheus и Grafana. Это дает воспроизводимое QA-окружение.

**Вопрос: Что значит container healthy?**  
Ответ: Healthcheck внутри контейнера успешно прошел. Например, сервис отвечает на `/ready`, PostgreSQL отвечает на `pg_isready`, Redis на `PING`.

**Вопрос: Что делать, если сервис не стартует?**  
Ответ: Проверить `docker compose ps`, затем `docker compose logs SERVICE`, проверить переменные окружения, порты, зависимости и healthcheck.

**Вопрос: Чем `docker compose down` отличается от `down -v`?**  
Ответ: `down` удаляет контейнеры и network, но оставляет volumes. `down -v` также удаляет volumes, значит базы и очереди будут очищены.

### Redis

**Вопрос: Как ты проверял Redis cache?**  
Ответ: Создавал продукт, дважды запрашивал его по ID и проверял `X-Cache`: первый запрос должен быть `MISS`, второй `HIT`. После изменения продукта кеш должен инвалидироваться.

**Вопрос: Что такое cache invalidation?**  
Ответ: Удаление или обновление устаревшей записи кеша после изменения исходных данных, чтобы клиент не получил старую информацию.

**Вопрос: Какие риски есть у кеширования?**  
Ответ: Устаревшие данные, сложность воспроизведения дефектов, зависимость от порядка запросов, скрытие ошибок БД, некорректная инвалидация.

### RabbitMQ и async

**Вопрос: Что такое exchange и queue?**  
Ответ: Exchange принимает сообщения и маршрутизирует их по правилам. Queue хранит сообщения до обработки consumer-ом.

**Вопрос: Что такое consumer?**  
Ответ: Компонент, который читает сообщения из очереди и обрабатывает их. В проекте notification-service является consumer-ом событий заказа.

**Вопрос: Почему notification может появиться не сразу?**  
Ответ: Потому что создание notification идет асинхронно через RabbitMQ. API создания заказа завершается раньше, чем consumer гарантированно обработает сообщение.

**Вопрос: Что такое eventual consistency?**  
Ответ: Состояние системы становится согласованным не мгновенно, а через некоторое время. Например, заказ уже создан, а notification появится после обработки события.

**Вопрос: Что такое idempotency?**  
Ответ: Повторная обработка одного и того же события не должна создавать дубликаты или ломать состояние. В проекте для этого используется уникальный `event_id`.

### Observability

**Вопрос: Что такое observability?**  
Ответ: Способность понимать состояние системы по внешним сигналам: logs, metrics, traces. В проекте есть JSON-логи, Prometheus metrics и Grafana dashboards.

**Вопрос: Зачем QA смотреть Grafana?**  
Ответ: UI/API может вернуть ответ, но метрики показывают скрытые проблемы: рост latency, 5xx, падение target, деградацию зависимости, рост request rate.

**Вопрос: Что такое Prometheus target?**  
Ответ: Endpoint, с которого Prometheus регулярно собирает метрики. В проекте это `/metrics` каждого микросервиса.

**Вопрос: Что такое counter?**  
Ответ: Метрика, которая только растет. Например, `http_requests_total`.

**Вопрос: Что такое histogram?**  
Ответ: Метрика, которая распределяет значения по buckets. Используется для расчета latency percentiles, например p95.

**Вопрос: Что такое p95 latency?**  
Ответ: 95% запросов быстрее или равны этому значению. Если p95 вырос, значит значительная часть пользователей испытывает замедление.

**Вопрос: Зачем нужен `X-Request-ID`?**  
Ответ: Он связывает события одного запроса в разных сервисах. По нему можно пройти путь запроса через order-service, product-service, payment-service и найти место сбоя.

### Resilience

**Вопрос: Как проверить отказ payment-service?**  
Ответ: Остановить контейнер `payment-service`, создать заказ, посмотреть response order-service, logs, readiness и Grafana. Затем запустить сервис обратно и проверить восстановление.

**Вопрос: Чем business failure отличается от technical failure?**  
Ответ: Business failure - ожидаемый отказ по бизнес-правилам, например declined payment. Technical failure - проблема инфраструктуры или зависимости, например timeout или сервис недоступен.

**Вопрос: Что такое timeout и почему он важен?**  
Ответ: Timeout ограничивает время ожидания зависимости. Без timeout один зависший сервис может задерживать или блокировать весь flow.

**Вопрос: Что такое graceful degradation?**  
Ответ: Система продолжает работать частично или возвращает контролируемую ошибку, а не падает полностью при отказе зависимости.

### Базы данных и данные

**Вопрос: Почему тестовые данные важны?**  
Ответ: Нестабильные или повторяющиеся данные дают ложные ошибки. Например, повторный email вызывает `409`, поэтому для позитивного теста нужен уникальный email.

**Вопрос: Как проверять консистентность данных?**  
Ответ: Сравнивать API responses, состояние связанных сущностей, events, notifications и логи. В микросервисах нельзя просто смотреть одну БД и считать систему проверенной.

**Вопрос: Почему прямой доступ к БД не всегда хорошая идея для тестирования?**  
Ответ: Он обходит публичный контракт сервиса. Для функционального тестирования главный источник истины - API. БД полезна для диагностики, но не должна заменять API-проверки.

### Собеседовательный рассказ о проекте

**Вопрос: Расскажи о самом сложном сценарии, который ты тестировал.**  
Ответ: Самый сложный сценарий - создание заказа. Он затрагивает user-service, product-service, payment-service, order-service, RabbitMQ и notification-service. Я проверял happy path, payment failure, payment timeout, недостаточный stock, неактивного пользователя, появление notification и метрики/логи по request ID.

**Вопрос: Как бы ты улучшил этот проект дальше?**  
Ответ: Добавил бы централизованные логи через Loki, tracing через OpenTelemetry, Sentry для exception tracking, более детальные бизнес-метрики, performance testing, security checks и только после сильной ручной базы начал бы API automation.

**Вопрос: Что ты вынес из проекта как QA?**  
Ответ: Я научился смотреть на систему шире одного endpoint: проверять контракты, зависимости, асинхронные события, кеш, отказоустойчивость, логи и метрики. Это приближает тестирование к реальным production-сценариям.
