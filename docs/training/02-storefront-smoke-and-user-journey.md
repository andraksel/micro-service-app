# Урок 02. Customer Store: путь реального пользователя

## Цель урока

Пройти приложение как обычный покупатель: открыть магазин, создать аккаунт, наполнить каталог, добавить товар в корзину, оформить заказ и увидеть статус.

## Компетенции после урока

После урока ты должен уметь:

- проверять пользовательский UI как black-box тестировщик;
- связывать экранные действия с backend API;
- понимать, какие сервисы участвуют в пользовательском journey;
- отличать frontend-дефект от backend-дефекта;
- формировать smoke checklist для e-commerce приложения;
- описывать пользовательский сценарий на собеседовании.

## Перед практикой: что нужно понять

В этом уроке ты тестируешь систему как обычный пользователь. Это называется black-box подход: ты смотришь на приложение через UI и сначала не лезешь в код.

Но сильный QA не останавливается на фразе "кнопка работает". Он понимает, какой backend-flow стоит за каждым действием.

### Что такое user journey

User journey - это путь пользователя в продукте от начала до результата.

Для магазина главный journey:

```text
Открыть магазин
  -> найти товар
  -> добавить товар в корзину
  -> создать профиль
  -> оформить заказ
  -> увидеть статус заказа
  -> получить уведомление
```

Этот путь важен, потому что он проверяет не отдельную кнопку, а ценность продукта. Если пользователь не может оформить заказ, магазин как бизнес-система не выполняет главную функцию.

### Какие слои участвуют в UI-сценарии

Когда ты работаешь с `Customer Store`, внутри участвуют:

| Действие в UI | Что происходит внутри |
| --- | --- |
| Открытие страницы | `nginx` отдает `store.html`, `store.css`, `store.js`. |
| Загрузка каталога | Browser вызывает `GET /api/products`; gateway отправляет запрос в `product-service`. |
| Seed demo catalog | Browser вызывает `POST /api/products` для демо-товаров. |
| Создание профиля | Browser вызывает `POST /api/users`; данные сохраняются в `user-db`. |
| Добавление в корзину | Обычно это frontend state/localStorage, backend еще не вызывается. |
| Checkout | Browser вызывает `POST /api/orders`; начинается интеграционный flow. |
| История заказов | Browser вызывает `GET /api/orders`; фильтрует заказы текущего пользователя. |
| Уведомления | Browser вызывает `GET /api/notifications?order_id=...`. |

### Что такое smoke test для UI

Smoke test - это короткая проверка, что приложение в принципе пригодно для дальнейшего тестирования.

Smoke для магазина не должен покрывать все edge cases. Он отвечает на вопрос:

```text
Можно ли сейчас тестировать стенд дальше, или он явно сломан?
```

Минимальный smoke для магазина:

- страница открывается;
- каталог загружается;
- есть товары или работает seed catalog;
- профиль создается;
- товар добавляется в корзину;
- checkout создает заказ;
- заказ отображается в истории;
- нет явных frontend errors.

### Как отличать frontend-дефект от backend-дефекта

Если кнопка не нажимается, не меняет состояние или текст на экране неправильный - это может быть frontend.

Если UI показывает ответ `404`, `422`, `500`, `502`, `503` - нужно смотреть API.

Практическое правило:

```text
Повтори действие напрямую через API.
```

Если API работает, а UI нет - вероятнее frontend-дефект.

Если API тоже падает - смотри gateway/backend/dependency.

### Какие дефекты реально можно найти в этом уроке

- UI вызывает неправильный endpoint;
- nginx route ведет не в тот сервис;
- кнопка checkout активна при пустой корзине;
- корзина считает total неправильно;
- UI теряет выбранного пользователя;
- заказ создан, но не отображается в истории;
- notification появляется, но UI ее не показывает;
- ошибка backend показывается пользователю непонятным текстом;
- поиск или фильтр скрывает товары некорректно;
- после refresh теряется состояние корзины или профиля.

### Что должен проговаривать QA

На каждом шаге спрашивай себя:

```text
Какое действие пользователя я проверяю?
Какой API должен быть вызван?
Какой сервис должен ответить?
Какие данные должны измениться?
Где я могу подтвердить результат?
Какой пользовательский impact, если это сломано?
```

## Что сделать

1. Открой магазин:

```text
http://localhost:8080/store.html
```

2. Убедись, что сверху виден статус `Store systems operational`.

3. В блоке `Catalog` нажми `Seed demo catalog`.

4. Проверь, что появились карточки товаров.

5. Используй поиск:

```text
Search: lamp
```

6. Сбрось поиск и проверь фильтр `Category`.

7. Добавь 2-3 товара в корзину.

8. Увеличь и уменьши количество товара в корзине.

9. Перейди в `Account`.

10. Создай аккаунт покупателя.

11. Вернись к корзине и нажми `Checkout`.

12. Перейди в `Orders`.

13. Проверь:

```text
order id
status
total amount
order items
notification events
```

## Куда смотреть и почему

Customer Store показывает внешний пользовательский слой. Он важен, потому что реальный пользователь не видит микросервисы, RabbitMQ, Redis и Grafana. Он видит только товары, корзину, checkout и статус заказа.

Если UI показывает ошибку, нужно понять, где проблема:

- не работает кнопка - возможно frontend event handler;
- `404/422/500` в сообщении - backend/API проблема;
- checkout зависает - возможно payment timeout или order-service;
- заказ создан, но нет notification - возможно RabbitMQ или notification-service;
- catalog пустой - возможно products отсутствуют или product-service недоступен.

## Что происходит внутри системы

`Seed demo catalog` создает товары через:

```text
POST /api/products
```

Создание аккаунта идет через:

```text
POST /api/users
```

Checkout идет через:

```text
POST /api/orders
```

Order-service внутри вызывает:

```text
user-service
product-service
payment-service
RabbitMQ
notification-service
```

## Проверки, которые нужно выполнить

| Проверка | Ожидаемый результат |
| --- | --- |
| Магазин открывается | `store.html` загружается без ошибок. |
| Seed catalog | Появляются товары. |
| Search | Список товаров фильтруется. |
| Category filter | Показываются товары выбранной категории. |
| Add to cart | Товар появляется в корзине. |
| Quantity increase/decrease | Total пересчитывается. |
| Create account | Account badge становится `active`. |
| Checkout | Создается заказ. |
| Orders | Заказ отображается в истории. |
| Notifications | Для заказа появляются event lines. |

## Вопросы на собеседовании

### Вопрос: Как бы ты протестировал интернет-магазин вручную?

Основной ответ:  
Я начал бы со smoke journey: открыть магазин, проверить каталог, поиск и фильтры, добавить товар в корзину, создать аккаунт, оформить заказ, проверить статус заказа и уведомления. Затем расширил бы проверки на негативные сценарии: пустая корзина, неактивный пользователь, недостаточный stock, payment failure, payment timeout, недоступность сервисов.

Альтернативный ответ:  
Сначала проверил бы критичный бизнес-flow покупки, затем детализацию по модулям: каталог, корзина, аккаунт, checkout, история заказов.

### Вопрос: Как понять, что ошибка в UI, а не в backend?

Основной ответ:  
Нужно сравнить поведение UI с прямым API-запросом. Если API работает, а UI нет, вероятна frontend-проблема. Если API тоже возвращает ошибку, нужно смотреть backend response, логи сервиса и gateway.

Альтернативный ответ:  
Проверяю DevTools/Network или curl/Postman. UI - это только один клиент к API.

### Вопрос: Что такое smoke test для магазина?

Основной ответ:  
Это минимальная проверка, что магазин пригоден для дальнейшего тестирования: главная страница открывается, каталог загружается, пользователь создается, товар добавляется в корзину, заказ оформляется, статус заказа отображается.

Альтернативный ответ:  
Быстрая проверка критичного пути покупки.

## Критерий завершения урока

Ты можешь оформить заказ через `Customer Store`, объяснить, какие API были вызваны, и показать этот заказ в `Orders`.
