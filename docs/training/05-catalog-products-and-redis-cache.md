# Урок 05. Каталог, товары, stock и Redis cache

## Цель урока

Научиться тестировать каталог товаров, остатки, фильтры, кеширование и инвалидацию кеша.

## Компетенции после урока

После урока ты должен уметь:

- проверять товарный каталог через UI и API;
- тестировать цену, stock, status и описание продукта;
- понимать `X-Cache: MISS` и `X-Cache: HIT`;
- объяснять, как Redis влияет на поведение чтения;
- проверять cache invalidation после изменения продукта;
- описывать риски кеширования на собеседовании.

## Перед практикой: что нужно понять

`product-service` отвечает за каталог товаров. Для магазина это витрина: пользователь выбирает товар, видит цену, остаток, описание, категорию и статус доступности.

Если каталог работает неправильно, пользователь может купить не тот товар, увидеть неверную цену, попытаться купить товар без остатка или не увидеть доступный товар.

### За что отвечает product-service

В этом проекте `product-service` отвечает за:

| Область | Что важно тестировать |
| --- | --- |
| Создание товара | Название, описание, цена, stock, category, image URL, status. |
| Список товаров | Карточки в магазине и ответ `GET /api/products`. |
| Конкретный товар | `GET /api/products/{id}` возвращает правильный товар. |
| Обновление товара | Изменение цены, stock, описания и статуса. |
| Soft delete товара | Товар переводится в `archived`. |
| Cache behavior | Первый запрос идет в БД, повторный может прийти из Redis. |

### Что такое stock

`stock` - это остаток товара на складе. Это не декоративное поле. От него зависит возможность заказа.

Примеры дефектов:

- UI показывает `5 in stock`, но API возвращает `0`;
- можно заказать больше, чем есть на складе;
- товар с `stock = 0` активен для checkout;
- после создания заказа stock не уменьшается, если продуктовая логика должна была это делать;
- фильтр или сортировка скрывает товары с положительным stock.

Даже если учебный проект не реализует полноценную складскую систему, QA должен мыслить как в реальном e-commerce: stock влияет на деньги, доверие и поддержку.

### Что такое Redis cache

Redis - это быстрое key-value хранилище в памяти. В проекте он используется как кеш для чтения конкретного товара.

Идея кеша:

```text
Первый запрос товара:
  product-service идет в PostgreSQL
  возвращает товар
  кладет копию в Redis
  отвечает X-Cache: MISS

Повторный запрос товара:
  product-service берет товар из Redis
  быстрее отвечает клиенту
  отвечает X-Cache: HIT
```

`MISS` не означает ошибку. Это значит, что в кеше еще не было записи.

`HIT` не означает, что данные точно правильные. Это значит, что сервис взял их из кеша.

### Почему кеш опасен для QA

Кеш ускоряет систему, но может показывать старые данные.

Типовой дефект:

```text
Цена товара была 15.50.
QA изменил цену на 21.00.
UI продолжает показывать 15.50.
API возвращает X-Cache: HIT.
```

В таком случае проблема может быть в cache invalidation: после изменения товара старая запись должна быть удалена или обновлена в Redis.

### Что такое cache invalidation

Cache invalidation - это сброс или обновление кеша, когда источник правды изменился.

Источник правды по товарам - PostgreSQL в `product-db`.

Redis - ускоритель чтения, но не главный источник правды.

Профессиональная формулировка:

```text
После PATCH /api/products/{id} следующий GET /api/products/{id} должен вернуть актуальные данные.
Если ответ приходит из Redis, кеш должен быть предварительно инвалидирован или обновлен.
```

### Что должен проговаривать QA

Перед проверкой каталога проговаривай:

```text
Я проверяю UI, API или кеш?
Какой сервис владеет данными товара?
Что является источником правды?
Что означает X-Cache в ответе?
После какого действия кеш должен сброситься?
Может ли пользователь увидеть устаревшую цену или stock?
```

Через Customer Store:

1. Открой `http://localhost:8080/store.html`.
2. Нажми `Seed demo catalog`.
3. Проверь карточки товаров.
4. Проверь поиск.

Через QA Dashboard:

1. Создай отдельный продукт.
2. Нажми `Check cache`.
3. Убедись:

```text

second HIT
```

Через API:

```powershell
Invoke-WebRequest -Uri http://localhost:8080/api/products/PRODUCT_ID -UseBasicParsing
Invoke-WebRequest -Uri http://localhost:8080/api/products/PRODUCT_ID -UseBasicParsing
```

Смотри header:

```text

```

## Cache testing

Проверка cache miss:

1. Создать новый продукт.
2. Первый раз запросить по ID.
3. Ожидать `X-Cache: MISS`.

Проверка cache hit:

1. Повторить GET по тому же ID.
2. Ожидать `X-Cache: HIT`.

Проверка invalidation:

1. Получить продукт дважды, чтобы был `HIT`.
2. Изменить продукт через `PATCH /api/products/{id}`.
3. Получить продукт снова.
4. Ожидать `MISS`, потому что старый кеш должен быть сброшен.

## Риски кеширования

- Пользователь может увидеть устаревшие данные.
- Баг может воспроизводиться только после первого запроса.
- Кеш может скрывать проблемы БД.
- Неверная invalidation приводит к старым ценам или stock.
- Cold cache может перегрузить БД после рестарта Redis.

## Практическое задание


Потом снова проверь товар:

```text

```

## Вопросы на собеседовании

### Вопрос: Что означает `X-Cache: MISS`?

Основной ответ:  
Это значит, что нужной записи не было в Redis, поэтому product-service взял данные из базы и положил их в кеш.

Альтернативный ответ:  
Данные пришли из primary storage, а не из кеша.

### Вопрос: Что означает `X-Cache: HIT`?

Основной ответ:  
Это значит, что product-service нашел товар в Redis и вернул его из кеша, не обращаясь к БД за основным чтением.

Альтернативный ответ:  
Сработал cached read.

### Вопрос: Почему cache invalidation сложна?

Основной ответ:  
Потому что после изменения данных нужно гарантированно убрать или обновить старую запись в кеше. Если этого не сделать, пользователи увидят устаревшие цены, описания или остатки.

Альтернативный ответ:  
Самая сложная часть кеша - не положить данные, а вовремя удалить старые.

### Вопрос: Как тестировать кеш?

Основной ответ:  
Нужно проверять последовательность запросов: первый read дает `MISS`, второй read дает `HIT`, update сбрасывает кеш, следующий read снова дает `MISS`. Также нужно тестировать поведение при недоступном Redis.

Альтернативный ответ:  
Проверяю headers, логи product-service и состояние Redis.
