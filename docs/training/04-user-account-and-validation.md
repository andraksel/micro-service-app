# Урок 04. User-service: аккаунты, валидация и состояние пользователя

## Цель урока

Научиться глубоко тестировать пользовательские данные: создание аккаунта, email uniqueness, статус пользователя и влияние аккаунта на checkout.

## Компетенции после урока

После урока ты должен уметь:

- тестировать поля формы и API-валидацию;
- проверять uniqueness constraint;
- понимать влияние состояния пользователя на бизнес-flow;
- оформлять баги по валидации;
- объяснять, почему тестовые данные должны быть уникальными;
- отличать UI validation от backend validation.

## Перед практикой: что нужно понять

`user-service` отвечает за пользователей. В реальном продукте это один из критичных сервисов, потому что без корректного пользователя нельзя оформить заказ, построить личный кабинет, применить ограничения, посмотреть историю покупок и связать действия с конкретным клиентом.

В этом проекте `user-service` хранит пользователя в своей базе `user-db`. Другие сервисы не должны напрямую лезть в эту базу. Они спрашивают пользователя через API.

### За что отвечает user-service

Основные зоны ответственности:

| Зона | Что проверяет QA |
| --- | --- |
| Создание пользователя | Валидный email и имя создают аккаунт со статусом `active`. |
| Уникальность email | Повторный email не создает второго пользователя. |
| Получение пользователя | По существующему `id` возвращаются корректные данные. |
| Статус пользователя | `active`, `blocked`, `deleted` влияют на checkout. |
| Soft delete | Удаление переводит пользователя в `deleted`, а не обязательно стирает запись физически. |
| Backend validation | API не принимает мусорные данные даже если UI ошибся. |

### Почему у сервиса своя база

В микросервисной архитектуре каждый сервис владеет своими данными. Это называется data ownership.

Для QA это означает:

```text
user-db - источник правды по пользователям.
order-service не должен сам решать, активен пользователь или нет.
order-service должен запросить user-service и использовать его ответ.
```

Если заказ оформляется для `blocked` пользователя, дефект может быть не только в `order-service`. Возможные причины:

- `user-service` неправильно сохранил статус;
- `order-service` не проверяет статус;
- gateway отправляет запрос не туда;
- UI показывает старый статус;
- тестировщик использует не тот `user_id`.

### Что такое статус пользователя

В проекте есть три статуса:

| Статус | Смысл | Как влияет на систему |
| --- | --- | --- |
| `active` | Пользователь нормальный и может покупать. | Checkout разрешен. |
| `blocked` | Пользователь существует, но не должен оформлять заказ. | Checkout должен быть отклонен. |
| `deleted` | Пользователь удален логически. | Checkout должен быть отклонен. |

Важно: статус - это не просто текст в карточке. Это бизнес-правило. Если UI показывает `blocked`, но заказ все равно создается, это дефект бизнес-логики.

### UI validation и backend validation

UI validation нужна для удобства пользователя: быстро подсветить ошибку в форме.

Backend validation нужна для безопасности и целостности данных. Пользователь может обойти UI через API, Postman, скрипт или сломанный frontend. Поэтому backend обязан сам проверять данные.

Пример:

```text
UI не дает ввести bad-email.
Но QA отправляет bad-email через API.
Backend обязан вернуть validation error.
```

Если backend принял `bad-email`, это серьезнее, чем дефект UI. Такой дефект позволяет записать некорректные данные в базу.

### Почему уникальные тестовые данные важны

Email уникален. Если тестировщик много раз использует `test@example.com`, второй и последующие тесты начнут падать с `409 Conflict`. Это не всегда баг приложения. Часто это плохая подготовка тестовых данных.

Профессиональный подход:

```text
manual-user-2026-06-06-001@example.com
qa-smoke-<timestamp>@example.com
duplicate-user@example.com только для проверки дубля
```

Уникальные данные позволяют отличать реальный дефект от мусора, накопленного в окружении.

### Что должен проговаривать QA

Перед проверкой пользователя проговаривай:

```text
Какие поля обязательны?
Какая validation должна сработать на UI?
Какая validation должна сработать на backend?
Какой статус пользователя ожидается?
Как этот статус должен повлиять на checkout?
Как я докажу, что проблема именно в user-service?
```

## Что сделать

Через Customer Store:

1. Открой `http://localhost:8080/store.html`.
2. Перейди в `Account`.
3. Создай аккаунт с валидным email.
4. Проверь, что badge стал `active`.
5. Попробуй создать аккаунт с тем же email через QA Dashboard или API.

Через API:

```powershell
$email = "duplicate-user@example.com"
$body = @{ email = $email; full_name = "Duplicate User" } | ConvertTo-Json
Invoke-WebRequest -Uri http://localhost:8080/api/users -Method POST -Body $body -ContentType "application/json"
Invoke-WebRequest -Uri http://localhost:8080/api/users -Method POST -Body $body -ContentType "application/json"
```

Ожидаемый результат:

```text
first request: 201
second request: 409
```

Проверь невалидные значения:

- пустой email;
- `bad-email`;
- пустой `full_name`;
- `full_name` из пробелов;
- слишком длинное имя.

## Куда смотреть и почему

Customer Store показывает, как ошибку видит пользователь.

QA Dashboard показывает raw JSON response и помогает быстрее понять backend result.

Swagger UI показывает ограничения схемы.

Логи user-service помогают понять, дошел ли запрос до backend:

```powershell
docker compose logs --tail=100 user-service
```

## Что важно понять

Backend должен валидировать данные даже если UI уже проверяет поля. UI validation улучшает UX, но не является защитой системы.

Email uniqueness - это бизнес- и data integrity правило. Если оно сломается, в системе появятся неоднозначные пользователи.

Статус пользователя влияет на order flow. Inactive/blocked/deleted user не должен успешно оформлять заказ.

## Практическое задание

1. Создай активного пользователя.
2. Создай продукт.
3. Оформи заказ - он должен пройти.
4. Через API измени статус пользователя на `blocked`:

```powershell
$body = @{ status = "blocked" } | ConvertTo-Json
Invoke-WebRequest -Uri http://localhost:8080/api/users/USER_ID/status -Method PATCH -Body $body -ContentType "application/json"
```

5. Попробуй оформить новый заказ с этим user id.

Ожидаемый результат:

```text
order-service returns business error: user is not active
```

## Артефакт урока

Составь таблицу:

| Поле | Валидное значение | Невалидные значения | Ожидаемый код | Риск |
| --- | --- | --- | --- | --- |
| email | `qa@example.com` | пусто, `bad-email`, duplicate | `201`, `422`, `409` | Дубли, невозможность идентификации. |

## Вопросы на собеседовании

### Вопрос: Почему backend validation обязательна, если есть UI validation?

Основной ответ:  
UI можно обойти через API, Postman, curl или другой клиент. Backend validation защищает систему и данные независимо от клиента.

Альтернативный ответ:  
Frontend validation - удобство пользователя, backend validation - целостность системы.

### Вопрос: Как тестировать уникальность email?

Основной ответ:  
Нужно дважды отправить запрос с одним email. Первый должен создать пользователя, второй должен вернуть `409 Conflict`. Также полезно проверить регистр email, например `Test@Example.com` и `test@example.com`, если система нормализует email.

Альтернативный ответ:  
Проверить duplicate create и убедиться, что в системе не появилось два пользователя с одним email.

### Вопрос: Что делать, если UI показывает общую ошибку, а API возвращает точную?

Основной ответ:  
Нужно завести frontend/UX defect или improvement: пользователь должен видеть понятную ошибку. При этом backend может работать корректно.

Альтернативный ответ:  
Локализую как проблему отображения ошибки на UI, если API contract корректен.

Ты можешь протестировать аккаунт на UI и API, объяснить ошибки `409/422`, и показать, как статус пользователя влияет на заказ.
