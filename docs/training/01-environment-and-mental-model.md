# Урок 01. Окружение и ментальная модель системы

## Цель урока

Научиться запускать весь стенд, понимать состав сервисов и объяснять архитектуру проекта простым техническим языком.

## Компетенции после урока

После урока ты должен уметь:

- запускать и останавливать весь проект через Docker Compose;
- проверять состояние контейнеров;
- отличать `Running` от `Healthy`;
- объяснять назначение каждого сервиса;
- понимать, что такое gateway, service-owned database, Redis, RabbitMQ, Prometheus и Grafana;
- объяснять, почему QA должен уметь смотреть не только UI, но и инфраструктурное состояние.

## Перед практикой: что нужно понять

Этот урок не про Docker-команды сами по себе. Главная цель - построить в голове карту системы. Когда у тебя есть карта, ты перестаешь хаотично проверять "работает / не работает" и начинаешь рассуждать как инженер:

```text
Какой пользовательский сценарий сломался?
Какой endpoint отвечает за этот сценарий?
Через какой gateway идет запрос?
Какой сервис должен обработать запрос?
Какая база или внешняя зависимость участвует?
Где я могу подтвердить проблему?
```

### Что такое микросервисное приложение

Монолит - это когда большая часть логики живет в одном приложении и часто в одной базе.

Микросервисное приложение - это когда система разделена на несколько сервисов. Каждый сервис отвечает за свою область.

В этом проекте магазин разделен так:

```text
users        -> user-service
products     -> product-service
orders       -> order-service
payments     -> payment-service
notifications -> notification-service
```

Зачем так делают:

- легче развивать отдельные части системы;
- легче локализовать дефекты;
- команды могут работать над разными сервисами;
- один сервис можно менять без полного переписывания системы;
- каждый сервис может иметь свою базу и свою внутреннюю модель данных.

Но у микросервисов есть цена:

- больше сетевых вызовов;
- больше мест, где может случиться timeout;
- сложнее отладка;
- нужно понимать зависимости;
- важно уметь смотреть логи, метрики, очереди и gateway.

### Что такое сервис в этом проекте

Сервис - это отдельное backend-приложение внутри Docker-контейнера. У каждого сервиса есть:

- свой API;
- свои endpoints;
- свои логи;
- свои метрики;
- своя зона ответственности;
- иногда своя база данных.

Ключевая мысль: сервис не должен отвечать "за все". Он отвечает за свою часть бизнес-домена.

| Сервис | Что делает простыми словами | Что проверять QA |
| --- | --- | --- |
| `user-service` | Хранит покупателей и их статус. | Создание пользователя, duplicate email, плохой email, active/blocked/deleted. |
| `product-service` | Хранит товары, цены, остатки и кеширует чтение через Redis. | Создание товара, валидация цены/stock, `X-Cache`, active/archived. |
| `order-service` | Главный оркестратор покупки. Собирает user/product/payment в один заказ. | Checkout, расчет суммы, статус заказа, ошибки зависимостей. |
| `payment-service` | Контролируемая платежная зависимость для тестов. | `success`, `failure`, `timeout`, `random`, влияние на заказ. |
| `notification-service` | Получает события заказов из RabbitMQ и сохраняет уведомления. | Появление уведомлений, задержки, idempotency, очередь сообщений. |

### Что такое nginx

`nginx` - это web server и reverse proxy.

В этом проекте он выполняет роль входной двери:

```text
Browser -> nginx -> backend service
```

Он делает две важные вещи:

1. Отдает статические файлы UI:

```text
store.html
store.css
store.js
index.html
app.js
styles.css
```

2. Направляет API-запросы в нужный сервис:

```text
/api/users         -> user-service
/api/products      -> product-service
/api/orders        -> order-service
/api/payments      -> payment-service
/api/notifications -> notification-service
```

Пример:

```text
Пользователь нажал "Создать профиль"
  -> browser отправил POST http://localhost:8080/api/users
  -> nginx получил запрос
  -> nginx отправил его в user-service:8000/users
  -> user-service создал пользователя
  -> ответ вернулся обратно через nginx в browser
```

### Что такое gateway

Gateway - это единая публичная точка входа в систему. В этом проекте gateway реализован через `nginx`.

Почему это удобно:

- frontend обращается к одному адресу `http://localhost:8080`;
- пользователю не нужно знать порты `8001`, `8002`, `8003`, `8004`, `8005`;
- внутренние сервисы можно спрятать внутри Docker-сети;
- можно централизованно настроить HTTPS, маршрутизацию, auth, rate limiting;
- QA может проверять систему так, как ее видит реальный клиент.

Почему gateway может быть источником дефекта:

- route может вести не в тот сервис;
- rewrite может обрезать путь неправильно;
- сервис может быть жив, но gateway не может до него достучаться;
- после пересоздания контейнеров gateway может использовать устаревший upstream, если DNS настроен неправильно;
- при падении backend-сервиса gateway может вернуть `502`, `503` или `504`.

Для тестировщика важно не говорить сразу "сломался backend". Иногда backend работает, а gateway route настроен неверно.

### Что такое Docker Compose в этом проекте

Docker Compose запускает все контейнеры из `docker-compose.yml`.

Когда ты выполняешь:

```powershell
docker compose up -d --build
```

Compose:

- собирает Docker images для сервисов;
- запускает PostgreSQL базы;
- запускает Redis;
- запускает RabbitMQ;
- запускает backend-сервисы;
- запускает nginx;
- запускает Prometheus и Grafana;
- создает общую Docker-сеть;
- подключает volumes для хранения данных.

Контейнеры внутри одной Docker-сети видят друг друга по именам сервисов:

```text
order-service может обратиться к user-service
product-service может обратиться к redis
notification-service может обратиться к rabbitmq
```

### Почему `Running` не равно `Healthy`

`Running` означает: процесс контейнера запущен.

`Healthy` означает: контейнер прошел healthcheck.

Пример:

```text
PostgreSQL container Running
```

Это значит, что процесс стартовал. Но база может еще не принимать подключения.

```text
PostgreSQL container Healthy
```

Это значит, что `pg_isready` прошел и база готова.

Для backend-сервиса:

```text
Running -> uvicorn process запущен
Healthy -> сервис ответил на readiness endpoint
```

Для QA это принципиально. Если тесты стартуют слишком рано, контейнеры могут быть `Running`, но сценарии еще будут падать, потому что система не готова.

### Что такое данные и где они живут

В проекте есть несколько типов данных:

| Данные | Где живут |
| --- | --- |
| Пользователи | `user-db` |
| Товары | `product-db` |
| Заказы | `order-db` |
| Уведомления | `notification-db` |
| Кеш товаров | `redis` |
| События заказов | `rabbitmq` |
| Метрики | `prometheus` |
| Dashboard-настройки Grafana | `grafana-data` volume |

Когда ты выполняешь:

```powershell
docker compose down
```

контейнеры останавливаются, но volumes остаются.

Когда ты выполняешь:

```powershell
docker compose down -v
```

удаляются volumes, то есть база, кеш, очереди и метрики сбрасываются.

### Что должен уметь объяснить QA после запуска

Не достаточно сказать:

```text
Я запустил docker compose, все работает.
```

Сильный ответ звучит так:

```text
Я поднял весь стенд через Docker Compose. Проверил контейнеры через docker compose ps и readiness через smoke. Понимаю, что nginx является gateway и проксирует /api/users, /api/products, /api/orders, /api/payments и /api/notifications в соответствующие сервисы. Каждый сервис отвечает за свою доменную область и часть данных. Для диагностики я могу смотреть logs, RabbitMQ, Redis behavior, Prometheus и Grafana.
```

## Что сделать

1. Открой PowerShell.

2. Перейди в проект:

```powershell
cd C:\microservices-qa-lab
```

3. Подними все сервисы:

```powershell
docker compose up -d --build
```

4. Проверь контейнеры:

```powershell
docker compose ps
```

5. Запусти smoke:

```powershell
py -3 scripts\fault_lab.py smoke
```

6. Открой основные инструменты:

```text
Customer Store: http://localhost:8080/store.html
QA Dashboard:   http://localhost:8080
Grafana:        http://localhost:3000 admin / admin
Prometheus:     http://localhost:9090
RabbitMQ:       http://localhost:15672 guest / guest
```

## Куда смотреть и почему

`docker compose ps`: показывает, какие контейнеры подняты и какие прошли healthcheck. Если контейнер `Running`, это значит процесс жив. Если `Healthy`, значит healthcheck сервиса прошел.

`fault_lab.py smoke`: проверяет пользовательски важные readiness endpoints. Это ближе к QA-проверке, чем просто `docker compose ps`.

`QA Dashboard`: нужен тестировщику для ручного управления системой: создать пользователя, продукт, заказ, переключить payment mode, посмотреть notifications.

`Customer Store`: нужен, чтобы почувствовать систему как реальный пользователь: витрина, корзина, аккаунт, checkout, история заказов.

`Grafana/Prometheus`: нужны для диагностики поведения после действий: выросла ли latency, есть ли ошибки, живы ли scrape targets.

`RabbitMQ`: нужен для проверки асинхронной части: сообщения, очереди, consumer, события заказа.

## Типовые ошибки новичка

- Смотреть только UI и не проверять `docker compose ps`.
- Путать `Running` и `Healthy`.
- Не понимать, что `nginx` - это gateway, а не backend-сервис.
- Думать, что если магазин открылся, то вся система полностью исправна.
- Не сохранять evidence при проблеме.

### Вопрос: Чем `Running` отличается от `Healthy`?

Основной ответ:  
`Running` означает, что процесс контейнера запущен. `Healthy` означает, что прошел healthcheck, например сервис ответил на `/ready`, PostgreSQL прошел `pg_isready`, Redis ответил на `PING`. Сервис может быть `Running`, но еще не готов обрабатывать рабочие запросы.

Альтернативный ответ:  
`Running` - состояние контейнера, `Healthy` - состояние приложения внутри контейнера.

Есть `user-service`, `product-service`, `order-service`, `payment-service`, `notification-service`, `nginx` gateway, PostgreSQL базы для сервисов, Redis, RabbitMQ, Prometheus и Grafana.

Альтернативный ответ:  
Это микросервисный магазин: отдельные сервисы для пользователей, товаров, заказов, платежей и уведомлений плюс инфраструктура для кеша, очередей и наблюдаемости.
