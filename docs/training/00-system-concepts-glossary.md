# Урок 00. Базовые понятия проекта простым языком

## Зачем нужен этот урок

Перед тем как тестировать микросервисное приложение, нужно понимать не только "куда нажать", но и что происходит внутри системы. Иначе тестировщик видит только UI и не может объяснить:

- почему заказ не создался;
- какой сервис сломался;
- почему UI открылся, но checkout не работает;
- почему уведомление приходит позже заказа;
- где искать evidence;
- чем проблема frontend отличается от проблемы backend;
- что отвечать на собеседовании, когда спрашивают про микросервисы.

Этот урок - словарь и карта проекта. Возвращайся к нему каждый раз, когда встречаешь незнакомый термин.

## Как думать о системе

Проект - это учебный интернет-магазин. Пользователь видит витрину, корзину, профиль и заказы. Но внутри это не один большой backend, а несколько маленьких сервисов.

Упрощенная цепочка:

```text
Browser
  -> nginx gateway
  -> нужный backend service
  -> база данных / Redis / RabbitMQ / другой service
  -> ответ обратно в Browser
```

Тестировщик должен уметь идти по этой цепочке в обе стороны:

- от UI-симптома внутрь системы;
- от backend-ошибки к пользовательскому impact.

Пример:

```text
Пользователь нажал "Оформить заказ"
  -> браузер отправил POST /api/orders
  -> nginx принял запрос
  -> nginx отправил запрос в order-service
  -> order-service проверил user-service
  -> order-service проверил product-service
  -> order-service вызвал payment-service
  -> order-service сохранил заказ в order-db
  -> order-service отправил событие в RabbitMQ
  -> notification-service прочитал событие
  -> notification-service сохранил уведомление
```

Если на любом шаге есть проблема, пользователь увидит ошибку, задержку, неправильный статус или отсутствие уведомления.

## Что такое frontend

Frontend - это часть приложения, которую видит пользователь в браузере.

В проекте есть два frontend-интерфейса:

- `Customer Store` - обычный магазин для пользователя;
- `QA Dashboard` - внутренний инструмент для тестировщика.

Frontend сам не хранит пользователей, товары и заказы. Он отправляет HTTP-запросы в backend API.

Пример:

```text
Кнопка "Создать профиль"
  -> POST /api/users
```

Что должен понимать QA:

- если кнопка не нажимается, это может быть frontend-дефект;
- если кнопка нажалась, но API вернул `422`, это может быть ошибка данных или backend validation;
- если API работает через Postman/curl, но не работает через UI, вероятно проблема в frontend;
- если API тоже падает, нужно смотреть gateway/backend/logs.

## Что такое backend service

Backend service - это отдельное приложение, которое отвечает за конкретную часть бизнес-логики.

В проекте есть пять основных backend-сервисов:

| Сервис | Простое объяснение | За что отвечает |
| --- | --- | --- |
| `user-service` | Сервис пользователей | Создает пользователей, хранит email, имя и статус. |
| `product-service` | Сервис каталога | Создает товары, хранит цену, описание, остаток, статус товара. |
| `order-service` | Сервис заказов | Собирает весь checkout-flow: пользователь, товары, платеж, заказ, события. |
| `payment-service` | Тестовый сервис платежей | Имитирует успешный платеж, отказ, timeout и случайное поведение. |
| `notification-service` | Сервис уведомлений | Получает события из RabbitMQ и сохраняет уведомления по заказу. |

Почему это важно:

- дефект нужно локализовать до конкретного сервиса;
- у каждого сервиса свой API, свои логи, свои метрики;
- в микросервисах проблема часто появляется на границе между сервисами.

## Что такое nginx

`nginx` - это легкий и быстрый web server / reverse proxy.

В этом проекте `nginx` делает две вещи:

1. Отдает статические UI-файлы:

```text
/index.html
/store.html
/store.js
/store.css
```

2. Проксирует API-запросы в нужные backend-сервисы:

```text
/api/users        -> user-service
/api/products     -> product-service
/api/orders       -> order-service
/api/payments     -> payment-service
/api/notifications -> notification-service
```

То есть браузер обращается к одному адресу:

```text
http://localhost:8080
```

А `nginx` уже решает, куда отправить запрос внутри Docker-сети.

## Что такое gateway

Gateway - это единая входная точка в систему.

Представь офисное здание:

- пользователь не ходит по кабинетам напрямую;
- он приходит на ресепшен;
- ресепшен направляет его к нужному отделу.

В проекте роль ресепшена выполняет `nginx`.

Пример:

```text
Browser -> http://localhost:8080/api/orders -> nginx -> order-service:8000/orders
```

Почему gateway полезен:

- пользователю не нужно знать адрес каждого сервиса;
- frontend работает с одним base URL;
- можно централизованно управлять маршрутизацией;
- можно добавить TLS/HTTPS, rate limiting, auth, compression;
- можно скрыть внутренние сервисы от прямого доступа.

Почему gateway важен для QA:

- ошибка может быть не в сервисе, а в маршруте gateway;
- если `/api/users` уходит не туда, UI будет получать странные ответы;
- если gateway не видит сервис в Docker-сети, появятся `502/503`;
- если rewrite настроен неправильно, endpoint может вернуть `404`.

## Что такое API

API - это способ, которым frontend и сервисы общаются друг с другом.

Пример API-запроса:

```text
POST /api/users
Content-Type: application/json

{
  "email": "qa@example.com",
  "full_name": "QA User"
}
```

Пример API-ответа:

```json
{
  "id": "uuid",
  "email": "qa@example.com",
  "full_name": "QA User",
  "status": "active"
}
```

QA проверяет:

- правильный endpoint;
- HTTP method;
- request body;
- response status;
- response body;
- headers;
- schema;
- business behavior.

## Что такое service-owned database

Service-owned database означает: каждый сервис владеет своей базой данных и не должен напрямую читать чужую базу.

В проекте:

```text
user-service -> user-db
product-service -> product-db
order-service -> order-db
notification-service -> notification-db
```

`order-service` не лезет напрямую в `user-db`. Он спрашивает `user-service` через API.

Почему так делают:

- сервисы слабее связаны между собой;
- каждый сервис отвечает за свои данные;
- легче менять внутреннюю схему базы;
- меньше риск, что один сервис сломает данные другого.

Что должен понимать QA:

- если пользователь не найден, смотреть надо `user-service`, а не `order-db`;
- если заказ создан, он хранится в `order-db`;
- если уведомления нет, смотреть надо `notification-service` и `notification-db`;
- прямой доступ к базе - это диагностический инструмент, а не основной путь проверки.

## Что такое Redis

Redis - это быстрое key-value хранилище в памяти. В проекте Redis используется как cache для товаров.

Cache нужен, чтобы не ходить в базу за одними и теми же данными каждый раз.

Пример:

```text
Первый запрос товара -> product-service идет в product-db -> X-Cache: MISS
Повторный запрос -> product-service берет из Redis -> X-Cache: HIT
```

Что проверяет QA:

- первый запрос дает `MISS`;
- повторный запрос дает `HIT`;
- после изменения товара cache должен обновиться или сброситься;
- если Redis недоступен, приложение должно вести себя контролируемо.

## Что такое RabbitMQ

RabbitMQ - это message broker, то есть посредник для сообщений между сервисами.

В проекте он нужен для асинхронных уведомлений.

Синхронно:

```text
service A вызывает service B и ждет ответ сразу
```

Асинхронно:

```text
service A публикует событие в RabbitMQ
service B забирает событие позже
```

Пример:

```text
order-service создал заказ
order-service отправил event order.created в RabbitMQ
notification-service прочитал event
notification-service создал notification
```

Почему это важно:

- заказ может быть уже создан, а уведомление появится через 1-2 секунды;
- это не всегда баг, это eventual consistency;
- если notification-service остановлен, события могут ждать в очереди;
- если RabbitMQ недоступен, асинхронная часть ломается.

## Что такое Prometheus

Prometheus - это система сбора метрик.

Метрики - это числовые показатели работы системы:

- сколько запросов пришло;
- сколько ошибок было;
- какая latency;
- жив ли сервис;
- сколько ответов `2xx/4xx/5xx`.

Сервисы отдают метрики на `/metrics`, а Prometheus регулярно их забирает.

QA использует Prometheus/Grafana, чтобы доказать не только "у меня ошибка", но и технический эффект:

```text
после payment timeout выросла latency order-service
после остановки сервиса выросли 5xx
после восстановления метрики вернулись к норме
```

## Что такое Grafana

Grafana - это интерфейс для графиков и dashboard.

Prometheus хранит метрики, Grafana красиво показывает их на графиках.

В проекте Grafana нужна, чтобы смотреть:

- request rate;
- latency;
- errors;
- readiness;
- общую картину по сервисам.

QA должен уметь связать действие с графиком:

```text
Я создал 5 заказов -> вырос request rate
Я включил payment-timeout -> выросла latency
Я остановил service -> появились ошибки
```

## Что такое health и ready

`/health` и `/ready` похожи, но отвечают на разные вопросы.

`/health`:

```text
Процесс жив?
```

`/ready`:

```text
Сервис готов принимать рабочие запросы?
```

Пример:

- сервис запустился, значит `/health` может быть `200`;
- но база еще недоступна, значит `/ready` должен быть неуспешным.

Для QA важнее `/ready`, потому что пользовательский сценарий зависит не только от процесса, но и от зависимостей.

## Что такое Docker Compose

Docker Compose - это способ запускать много контейнеров одной командой.

Вместо того чтобы вручную запускать:

- `user-service`;
- `product-service`;
- `order-service`;
- PostgreSQL;
- Redis;
- RabbitMQ;
- Prometheus;
- Grafana;

ты запускаешь:

```powershell
docker compose up -d --build
```

Compose читает `docker-compose.yml` и поднимает всю систему.

Что должен понимать QA:

- контейнер - это изолированный процесс;
- сервисы внутри Docker общаются по service name;
- данные баз лежат в Docker volumes;
- `docker compose down -v` удаляет volumes и сбрасывает данные;
- `docker compose ps` показывает состояние контейнеров;
- `docker compose logs` помогает искать ошибки.

## Главная ментальная модель QA

Когда что-то сломалось, не угадывай. Иди по цепочке:

1. Что видит пользователь в UI?
2. Какой запрос ушел из UI?
3. Какой status code вернул API?
4. Какой route обработал gateway?
5. Какой backend-service отвечает за это поведение?
6. Есть ли ошибка в логах сервиса?
7. Жива ли база/Redis/RabbitMQ?
8. Есть ли метрика в Grafana?
9. Какой impact для пользователя?
10. Какой root cause и какой fix?

Если ты можешь пройти эту цепочку, ты уже думаешь как сильный QA, а не как человек, который просто нажимает кнопки.

## Мини-шпаргалка

| Термин | Простое объяснение |
| --- | --- |
| Frontend | То, что пользователь видит в браузере. |
| Backend | Серверная логика, которая обрабатывает данные. |
| Service | Отдельная часть backend, отвечающая за свою зону. |
| API | Правила общения между frontend и backend или между сервисами. |
| Gateway | Единая точка входа, которая направляет запросы в сервисы. |
| nginx | Web server / reverse proxy, который в проекте работает gateway. |
| PostgreSQL | Реляционная база данных. |
| Redis | Быстрый cache в памяти. |
| RabbitMQ | Очередь сообщений для асинхронного обмена. |
| Prometheus | Сборщик метрик. |
| Grafana | Dashboard для метрик. |
| Healthcheck | Проверка, что контейнер/сервис жив и готов. |
| Fault injection | Намеренное внесение поломок для тренировки диагностики. |

## Критерий завершения урока

Ты можешь без подсказки объяснить:

- зачем нужен `nginx`;
- что такое gateway;
- за что отвечает каждый сервис;
- чем Redis отличается от PostgreSQL;
- зачем нужен RabbitMQ;
- чем Prometheus отличается от Grafana;
- почему `/ready` важнее простого факта, что контейнер `Running`;
- как пользовательский клик проходит через систему.
